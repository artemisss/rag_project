# AGENT.md

## Роль агента

Этот репозиторий развивает `ReviewOps AI` — внутренний AI-сервис для автоматической обработки отзывов с маркетплейсов. Любой агент или разработчик, работающий в проекте, должен считать продуктовым source of truth `README.md` и реализовывать систему строго вокруг сценария customer care для Ozon, Wildberries и Яндекс.Маркета.

## Что строим

Система должна:

- забирать новые неотвеченные отзывы из marketplace API;
- нормализовывать их в единую модель;
- доставать релевантные policies и golden examples;
- вызывать LLM через OpenAI Responses API;
- принимать решение `auto publish`, `manual review` или `escalation`;
- публиковать ответ обратно либо показывать его в очереди модерации;
- хранить аудит, версии промптов, историю статусов и аналитику.

## Что нельзя делать

- нельзя придумывать endpoint-ы, payload-ы или ограничения Ozon;
- нельзя генерировать свободный текст там, где нужен строгий JSON;
- нельзя привязывать бизнес-логику к UI;
- нельзя смешивать retrieval, LLM и policy engine в один неразделимый модуль;
- нельзя отправлять marketplace secrets или сырые токены в логи;
- нельзя полагаться на память модели вместо базы знаний;
- нельзя публиковать рискованные ответы в обход policy engine;
- нельзя записывать дубли отзывов при повторном sync.

## Архитектурные инварианты

### 1. Единая модель отзыва

Все коннекторы должны приводить данные к общей сущности `UnifiedReview` или эквивалентной ORM-модели `reviews`.

Обязательный уникальный ключ:

```text
(marketplace, marketplace_account_id, external_review_id)
```

### 2. Двухшаговый LLM pipeline

LLM работает в двух отдельных фазах:

1. классификация;
2. генерация ответа.

Фазы не смешиваются, потому что:

- classification проще тестировать;
- policy engine получает прозрачные reason codes;
- generation зависит от strategy и retrieval;
- проще логировать ошибки и деградацию качества.

### 3. RAG вместо разросшегося prompt

В проекте нельзя хранить все правила и примеры внутри одного системного промпта. Используем:

- короткий system prompt;
- dynamic business context;
- retrieved context из `knowledge_items`.

Для MVP retrieval строится на `SQLite FTS5 + metadata filters`.

### 4. Policy engine обязателен

Модель не принимает финальное решение о публикации. Финальное решение принимает backend policy engine на основе:

- рейтинга;
- тематики;
- risk level;
- confidence score;
- brand policies;
- auto-reply rules;
- специальных исключений.

### 5. Auditability by default

Все важные действия должны быть объяснимы и воспроизводимы:

- какой отзыв пришел;
- какая версия промпта использовалась;
- какие knowledge items были выбраны;
- что вернула модель;
- почему система решила auto/manual/escalate;
- был ли ответ опубликован и с каким результатом.

## Структура проекта

Рекомендуемая структура:

```text
app/
  api/
  core/
  db/
  schemas/
  services/
    connectors/
    llm/
    knowledge/
    policy/
    reviews/
  jobs/
  admin/
tests/
docs/
```

Правила владения слоями:

- `api/` только принимает запросы и отдает ответы;
- `services/` содержит бизнес-логику;
- `connectors/` инкапсулируют интеграции;
- `db/` описывает модели, сессии и миграции;
- `schemas/` определяют входные и выходные контракты;
- `jobs/` запускают sync и фоновые процессы.

## Технологические решения

Базовый стек проекта:

- `FastAPI`
- `SQLAlchemy`
- `SQLite` для MVP
- `Alembic`
- `APScheduler`
- `httpx`
- `Pydantic`
- `OpenAI Responses API`

При росте нагрузки допустима миграция на:

- `PostgreSQL`
- `Redis`
- отдельные workers

Но до этого нельзя усложнять проект преждевременно.

## Правила по маркетплейсам

### Wildberries

- использовать отдельный rate limiter;
- опрашивать count unanswered перед полной выборкой;
- учитывать pagination и безопасные retry;
- отправку ответа делать отдельным publish-шагом.

### Яндекс.Маркет

- webhook является основным режимом;
- polling остается fallback-механизмом;
- webhook-обработчик должен быть быстрым;
- по событию сначала подтверждаем прием, затем дотягиваем полный payload в backend-процессе.

### Ozon

- использовать только adapter contract;
- реальные маршруты фиксировать только после верификации seller API;
- до подтверждения интеграции держать реализацию через интерфейсные заглушки и тестовые моки.

## Правила по LLM

- использовать `gpt-5.4` через Responses API, если иное явно не пересмотрено;
- использовать structured outputs для classification и generation;
- устанавливать `store: false`;
- по возможности удалять или маскировать PII перед отправкой;
- логировать `model_name`, `prompt_version_id`, токены, latency и результат;
- не разрешать модели самостоятельно решать вопрос публикации в обход policy engine.

## Правила по знаниям

В `knowledge_items` храним два типа:

- `policy`
- `example`

Каждый элемент должен иметь метаданные, позволяющие retrieval:

- marketplace;
- brand_id;
- issue_type;
- rating_bucket;
- language;
- priority;
- is_active.

Если ответ вручную признан особенно удачным, его нужно уметь переводить в `golden example`.

## Правила по безопасности

- ключи маркетплейсов хранить в зашифрованном виде;
- токены не показывать полностью в UI;
- чувствительные поля не писать в stdout-логи;
- любые действия оператора писать в `audit_log`;
- разграничивать роли `Admin`, `Support Manager`, `Content Manager`, `Analyst`.

## Правила по API

Минимальный набор endpoint-ов должен покрывать:

- integrations;
- reviews;
- knowledge;
- prompts;
- webhooks.

Каждый endpoint обязан:

- использовать Pydantic-схемы;
- иметь предсказуемые коды ответа;
- писать диагностируемые ошибки;
- не ломать идемпотентность sync-процессов.

## Правила по UI и админке

Админка должна быть операционной панелью, а не декоративной витриной.

Обязательные UX-принципы:

- оператор видит причину решения системы;
- из карточки отзыва доступны raw payload, найденные policies и used knowledge;
- рискованные кейсы отделены в moderation queue;
- bulk actions поддерживаются для типовых кейсов;
- экран Prompts & Model содержит sandbox для тестов.

Если в репозитории есть старые визуальные концепты, противоречащие этому продукту, они не являются обязательным source of truth.

## Порядок разработки

Работать стоит в таком порядке:

1. foundation backend;
2. модели БД и миграции;
3. CRUD и review pipeline;
4. knowledge retrieval;
5. OpenAI integration;
6. policy engine;
7. marketplace connectors;
8. admin UI;
9. analytics и hardening.

Не стоит начинать с красивой админки без работающего review pipeline.

## Тестовая стратегия

Минимально нужны:

- unit-тесты policy engine;
- unit-тесты normalizer и status transitions;
- contract-тесты connector adapters;
- integration-тесты review pipeline;
- тесты идемпотентности sync;
- тесты rollback и ошибок публикации.

Для LLM-слоя нужно тестировать:

- схемы structured output;
- устойчивость к пустым и токсичным отзывам;
- корректную маршрутизацию risky кейсов;
- стабильность reason codes.

## Definition of Done для задач

Любая задача считается завершенной, если:

- поведение реализовано в нужном слое;
- инварианты домена не нарушены;
- новые статусы и переходы отражены в `review_events`;
- при необходимости обновлены `README.md` и этот `AGENT.md`;
- добавлены или обновлены тесты;
- логирование и ошибки пригодны для диагностики.

## Когда нужно остановиться и уточнить решение

Нужно остановиться и поднять вопрос, если:

- marketplace API не подтвержден документально;
- требуется обойти policy engine ради скорости;
- предлагается хранить knowledge только в prompt;
- реализация ломает объяснимость решений;
- новая идея требует тяжелой инфраструктуры без необходимости для MVP.

## Короткое правило

Проект должен оставаться простым, объяснимым и управляемым: `ingest -> normalize -> retrieve -> classify -> generate -> decide -> publish/moderate -> audit`.
